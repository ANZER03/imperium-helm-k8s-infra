# Helm Runbook

This runbook outlines the essential Helm commands used to manage repositories, subcharts, deployments, and troubleshooting in the **Imperium Kubernetes Infrastructure** project.

---

## 1. Repository Management

To retrieve upstream charts (like Spark Operator, Strimzi Kafka, etc.), manage Helm repositories:

### Add a Repository
```bash
helm repo add <repo-name> <repo-url>
```
*Example:*
```bash
helm repo add strimzi https://strimzi.io/charts
helm repo add spark-operator https://kubeflow.github.io/spark-operator
```

### Update Repositories
Update local repository cache to fetch the latest chart indexes:
```bash
helm repo update
```

### List Repositories
List all repositories configured locally:
```bash
helm repo list
```

## 2. Dependency & Local Sub-chart Management

When working with an umbrella chart containing local subcharts (e.g., `pg-cluster`, `UI`, `kafka-connect`), updates to subcharts are not automatically picked up if packaged `.tgz` dependencies are stale.

### Update Dependencies (Re-package local charts)
Updates `Chart.lock` and compiles local directories into `.tgz` archives inside the `charts/` folder:
```bash
helm dependency update ./imperium-streaming-news-processing
```
> [!IMPORTANT]
> Run this command whenever you change values or templates inside any sub-chart (e.g., `charts/pg-cluster/values.yaml`) before applying the deployment.

### Build Dependencies
Recreates the `charts/` directory based on the existing `Chart.lock` file:
```bash
helm dependency build ./imperium-streaming-news-processing
```

### List Dependencies
Verify the status of all subcharts configured in `Chart.yaml`:
```bash
helm dependency list ./imperium-streaming-news-processing
```

## 3. Installation & Upgrades

Deploy or update the application stacks in the cluster:

### Install a Chart (Fresh Deploy)
```bash
helm install <release-name> <chart-path> --namespace <namespace> --create-namespace
```

### Upgrade / Install (Recommended)
Upgrades a release if it exists, or installs it if it does not.
```bash
helm upgrade --install <release-name> <chart-path> --namespace <namespace>
```
*Example (Deploying the main stack):*
```bash
helm upgrade --install imperium-streaming-news-processing ./imperium-streaming-news-processing -n imperium-news-ns
```

### Apply with custom values override
Pass explicit custom override files or parameters:
```bash
helm upgrade --install <release-name> <chart-path> -f <values-file.yaml> -n <namespace>
```

## 4. Deletion & Cleanup

Uninstall charts and clear resources from the Kubernetes cluster.

### Uninstall a Release
This will delete all Kubernetes resources (Deployments, Services, ConfigMaps, etc.) associated with the release:
```bash
helm uninstall <release-name> --namespace <namespace>
```
*Example:*
```bash
helm uninstall imperium-streaming-news-processing -n imperium-news-ns
```

> [!WARNING]
> CNPG `Cluster` resources and their PVCs (like `imperium-pg-1`) might not be fully deleted or get stuck in `Terminating` if pods are active or storage class constraints prevent resizing. If resources get stuck, clean them up manually using `kubectl`:
> ```bash
> kubectl delete cluster imperium-pg -n imperium-news-ns
> kubectl delete pvc imperium-pg-1 -n imperium-news-ns
> ```

## 5. Inspection, Templating & Troubleshooting

Debug what values and manifests are generated by Helm.

### Render Templates (Dry Run / Debug manifests)
Generates the raw Kubernetes YAML manifests locally without deploying them to verify formatting and syntax:
```bash
helm template <release-name> <chart-path>
```
*Filter by a specific template file:*
```bash
helm template imperium-streaming-news-processing ./imperium-streaming-news-processing -s charts/pg-cluster/templates/cluster.yaml
```

### Get Active Values
Check what values are currently deployed in the active release:
```bash
helm get values <release-name> -n <namespace>
```

### List Releases
List all active deployments in a namespace (or all namespaces with `-A`):
```bash
helm list -n <namespace>
# Or
helm list -A
```

### Retrieve Status
Check the status of a deployed release:
```bash
helm status <release-name> -n <namespace>
```

### Rollback a Release
Roll back to a previous deployment revision if an upgrade fails:
```bash
helm rollback <release-name> <revision-number> -n <namespace>
```

## 6. Managing Custom Images and Registry

When building and pushing custom images (like `llama-cpp` containing the embedded model), you must interact with the local registry.

### Port-forward the Registry
Since the registry service is inside the K3s cluster, port-forward it to expose it on the host:
```bash
kubectl port-forward svc/registry -n registry 30500:5000 &
```

### Build the Custom Image
Build the Docker image containing the downloaded model:
```bash
docker build -t localhost:30500/llama-cpp-gemma:latest -f DockerFiles/llama-cpp/Dockerfile .
```

### Push to local registry
Push the image to the local registry:
```bash
docker push localhost:30500/llama-cpp-gemma:latest
```
